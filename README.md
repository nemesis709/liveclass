# LiveClass Weekly Study Planner

## 프로젝트 개요

한 주간의 학습 일정을 시간표 형태로 편집하고 저장하는 학습 플래너입니다.  
수강생은 30분 단위 그리드에서 학습 블록을 추가, 수정, 삭제할 수 있고, 저장 전까지의 변경 사항은 저장본과 분리된 draft 상태로 관리됩니다.

이번 과제에서는 다음 세 가지를 핵심 문제로 보고 구현했습니다.

- 시간 기반 UI와 30분 단위 위치 계산
- 시간 충돌 감지와 저장 차단
- 서버 상태와 편집 중 상태의 분리

## 기술 스택

- TypeScript
- React 18
- Next.js 15 App Router
- TanStack Query
- Axios
- Vitest
- Testing Library

Mock API는 Next.js Route Handler로 구성했습니다.  
이 방식을 선택한 이유는 별도 서버 없이도 브라우저에서 바로 실행 가능하고, 저장본 API와 draft API를 함께 시뮬레이션하기 쉬웠기 때문입니다.

## Mock 데이터 / 서버 구성 방법

Mock API는 Next.js Route Handler로 구성했습니다.  
`pnpm dev` 실행 시 별도 mock 서버를 띄우지 않아도 아래 API가 함께 동작합니다.

- `GET /api/courses`
- `GET /api/planner`
- `PUT /api/planner`
- `GET /api/planner/draft`
- `PUT /api/planner/draft`
- `DELETE /api/planner/draft`

Mock 원본 데이터와 상태 관리는 아래 파일에 있습니다.

- `src/mocks/data/courses.ts`
- `src/mocks/data/planners.ts`
- `src/mocks/handlers/plannerMockStore.ts`

주의:

- mock store는 메모리 기반이라 dev 서버를 재시작하면 데이터가 초기화됩니다.

## 프로젝트 구조 설명

레이어 최상위 구조를 기준으로 두고, 각 레이어 내부에서 `planner`, `course` 단위로 나눴습니다.

```text
src/
  app/                    # Next.js App Router, API route, 전역 스타일
  model/                  # API 모델, draft 모델
  repository/             # Axios 기반 API 호출 계층
  view-model/             # TanStack Query 기반 화면 상태 계층
  view/                   # 화면 컴포넌트
  lib/                    # 시간/충돌/summary 유틸
  mocks/                  # Mock 데이터와 mock store
  shared/                 # 공통 API 클라이언트, query key

docs/
  design/spec.md          # 설계 문서
  api/swagger.yaml        # API 명세
  database/erd.md         # 데이터 구조 문서
```

## 요구사항 해석 및 가정

- 기본 시간 범위는 `08:00 ~ 20:00`입니다.
- 가산점 항목을 반영해 `30분 단위`를 기본 지원합니다.
- 시간 충돌은 실제 시간이 겹치는 경우만 충돌로 간주합니다.
  - `09:00-10:00`과 `10:00-11:00`은 충돌 아님
  - `09:00-10:00`과 `09:30-10:30`은 충돌
- API 스키마는 과제 명세를 따라 `HH:mm` 문자열을 유지합니다.
- 내부 계산은 문자열 비교 대신 분 단위 정수로 변환해 처리합니다.
- 충돌이 있는 블록도 draft에는 저장할 수 있게 두고, 최종 저장만 차단합니다.
  - 사용자가 충돌 상태를 실제 화면에서 확인하고 수정할 수 있도록 하기 위한 선택입니다.
- 주간 요약은 저장 데이터의 파생 정보라고 판단해 서버 응답 `summary` 필드로 함께 제공합니다.
- draft는 주차별 서버 리소스로 관리하며, 최종 저장 성공 시 정리합니다.

## 설계 결정과 이유

### 1. 저장본과 draft 분리

저장된 플래너는 TanStack Query로 관리하고, 편집 중 상태는 draft API를 통해 별도로 저장합니다.  
이 구조를 통해 저장 실패 시에도 편집 상태를 잃지 않고, 재진입 시 draft를 복구할 수 있습니다.

### 2. 시간 데이터의 이중 표현

외부 API는 `HH:mm` 문자열을 유지하되, 내부 로직은 분 단위 정수로 변환합니다.  
덕분에 충돌 판정, duration 계산, 그리드 위치 계산을 더 단순하고 안전하게 처리할 수 있습니다.

### 3. 충돌 허용 + 최종 저장 차단

처음에는 충돌 블록 자체를 입력 단계에서 막는 방식도 고려했지만, 이 경우 사용자가 실제 충돌 상태를 화면에서 확인할 수 없었습니다.  
현재 구현에서는 draft 단계에서는 충돌을 허용하고, 시각적으로 강조한 뒤 최종 저장만 차단합니다.

### 4. 요약의 서버 책임

총 학습 시간, 강의별 집계, 요일별 집계는 파생 정보이므로 여러 클라이언트에서 같은 규칙을 공유하는 편이 낫다고 판단했습니다.  
따라서 mock API에서도 `summary`를 응답에 포함하도록 확장했습니다.

### 5. 모바일 일별 뷰

모바일에서 주간 전체 그리드를 그대로 축소하면 가독성이 크게 떨어집니다.  
그래서 모바일에서는 요일 하나만 선택해서 보는 일별 뷰로 단순화했고, 그리드 상단 요일 헤더를 눌러 선택 모달을 여는 방식으로 UX를 구성했습니다.

### 6. 드래그 편집 전략

블록 이동은 클릭 기반 모달 편집만으로도 요구사항은 충족할 수 있었지만, 시간 여유가 있어 데스크톱/모바일 모두에서 드래그 편집을 추가했습니다.  
데스크톱은 시간과 요일을 함께 바꿀 수 있도록 가로/세로 이동을 모두 허용했고, 모바일은 오조작 가능성을 줄이기 위해 세로 이동만 허용했습니다.

드래그 중에는 실제 draft를 매 프레임 저장하지 않고 preview 블록만 갱신하고, 드롭 시점에만 draft 저장을 수행합니다.  
또한 드래그 시작 시 포인터가 블록 내부 어디를 잡았는지 `offset`을 저장해, 블록 상단 기준이 아니라 사용자가 잡은 지점이 유지된 채 따라오도록 보정했습니다.

## 구현 내용

- 주간 플래너 초기 조회
- 강의별 색상 블록 배치
- 블록 추가 / 수정 / 삭제 모달
- 30분 단위 그리드 및 위치 계산
- 블록 드래그 이동
  - 데스크톱: 요일/시간 동시 이동
  - 모바일: 같은 요일 내 시간 이동
- draft API 조회 / 저장 / 삭제
- 최종 저장 버튼과 dirty / loading / success / error 상태
- 충돌 블록 시각화 및 최종 저장 차단
- 이전 주 / 다음 주 이동
- 미저장 변경 경고
- `beforeunload` 기반 이탈 방지
- 모바일 일별 선택 뷰
- 핵심 로직 및 화면 흐름 테스트

## 미구현 / 제약사항

- 차트 기반 요약 시각화는 구현하지 않았습니다.
- draft autosave 주기, 만료 정책, 다중 기기 동기화 정책은 단순화했습니다.
- mock store는 메모리 기반이라 dev 서버를 재시작하면 데이터가 초기화됩니다.
- 블록 리사이즈 기반 편집은 지원하지 않고, 현재는 이동 중심의 드래그 편집만 제공합니다.
