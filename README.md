# LiveClass Weekly Study Planner

## 프로젝트 개요

LiveClass Weekly Study Planner는 수강생이 한 주간의 학습 일정을 시간표 형태로 편집하고 저장할 수 있도록 돕는 학습 플래너입니다. 사용자는 30분 단위 시간 그리드에서 학습 블록을 추가, 수정, 삭제할 수 있으며, 저장 전까지의 변경 사항은 별도 draft 상태로 유지됩니다.

본 과제는 시간 기반 UI, 시간 충돌 감지, 저장된 서버 상태와 편집 중 상태의 분리, 주간 요약 제공을 핵심 문제로 보고 구현합니다.

## 기술 스택

- TypeScript
- React
- Next.js App Router
- TanStack Query
- Mock API
  - 구현 예정: Next.js Route Handler 또는 MSW 기반 로컬 Mock

## 실행 방법

구현 후 아래 방식으로 로컬 실행할 예정입니다.

```bash
npm install
npm run dev
```

기본 실행 주소:


## 프로젝트 구조 설명

현재 프로젝트는 레이어 최상위 구조를 기준으로 정리하고, 각 레이어 내부에서 feature 단위로 구분합니다.

```text
src/
  app/                    # Next.js App Router 진입점 및 라우트
  model/
    planner/             # 플래너 API 모델, draft 모델
    course/              # 강의 참조 모델
  repository/
    planner/             # 플래너 API 호출 계층
    course/              # 강의 목록 API 호출 계층
  view-model/
    planner/             # 서버 상태와 draft 상태를 연결하는 화면 상태 계층
  view/
    planner/             # 플래너 화면 컴포넌트
  lib/
    planner/             # 플래너 전용 로직
    time/                # 시간 파싱, 포맷, 충돌 판정 유틸
  mocks/                 # Mock API 데이터 및 핸들러
  shared/                # 공통 UI, API 클라이언트, 유틸

  design/spec.md         # One Pager 형식의 설계 문서
  api/swagger.yaml       # API 명세
  database/erd.md        # 데이터 구조 문서
```

## 요구사항 해석 및 가정

- 플래너 기본 시간 범위는 `08:00 ~ 20:00` 입니다.
- 시간 슬롯은 가산점 항목을 반영해 `30분 단위`를 기본 지원합니다.
- 시간 충돌은 실제 시간이 겹치는 경우만 충돌로 간주합니다.
  - 예: `09:00-10:00`과 `10:00-11:00`은 충돌 아님
  - 예: `09:00-10:00`과 `09:30-10:30`은 충돌
- API 스키마는 과제 명세를 따르기 위해 `HH:mm` 문자열을 유지합니다.
- 내부 시간 계산은 문자열 비교 대신 `분 단위 정수`로 변환해 처리합니다.
- 저장 전 수정 사항은 최종 저장본과 분리된 draft API를 통해 별도로 관리합니다.
- 주간 요약은 클라이언트 임시 계산이 아니라 서버 책임으로 판단하여 `PlannerResponse.summary` 확장 필드로 내려주는 구조를 채택했습니다.
- draft는 주차별 서버 리소스로 관리하며, 재진입 시 draft가 존재하면 저장본보다 우선 복구합니다.
- 대신 가산점 항목 중 `30분 단위`, `이전 주/다음 주 이동`, `미저장 변경 경고`, `브라우저 이탈 방지`를 우선 구현 대상으로 둡니다.

## 설계 결정과 이유

### 1. 서버 상태와 편집 상태 분리

서버에 저장된 플래너와 사용자가 편집 중인 draft를 분리 관리합니다. 조회 결과는 TanStack Query로 관리하고, 편집 중인 블록은 `planner draft` API를 통해 주차별로 저장 및 복구합니다. 이 구조를 통해 새로고침이나 재진입 이후에도 수정 내역을 유지할 수 있고, 저장 실패 시에도 사용자의 편집 흐름을 보존할 수 있습니다.

### 2. 시간 데이터의 이중 표현

외부 API는 과제 명세를 존중해 `HH:mm` 문자열을 사용합니다. 반면 내부 로직은 시간을 분 단위 정수로 변환해 충돌 판정, duration 계산, 30분 단위 그리드 위치 계산에 사용합니다. 이를 통해 문자열 비교에 의존하지 않고 더 안정적인 시간 로직을 구성합니다.

### 3. 주간 요약의 서버 책임화

총 학습 시간, 강의별 집계, 요일별 집계는 저장 데이터의 파생 정보이며, 여러 플랫폼에서 동일한 규칙을 공유해야 하는 영역이라고 판단했습니다. 따라서 요약 계산 책임은 서버에 두고, 조회 및 저장 응답에 `summary`를 함께 포함하는 구조로 설계했습니다.

### 4. 레이어 최상위 구조

이번 과제는 실질적으로 `planner` 중심의 단일 기능에 가깝기 때문에, 도메인 최상위 구조보다 레이어 최상위 구조가 더 단순하고 설명 가능성이 높다고 판단했습니다. 대신 각 레이어 내부에서는 `planner`, `course`처럼 feature 하위 폴더를 두어 확장성을 확보합니다.

### 5. Draft API 채택

임시저장 상태를 localStorage에만 두는 방식도 가능하지만, 실제 서비스 관점에서는 저장본과 draft를 모두 서버 리소스로 관리하는 편이 더 일관적이라고 판단했습니다. 따라서 `GET/PUT /api/planner/draft`를 통해 주차별 draft를 조회하고 저장하며, 최종 저장 성공 시 draft를 정리하는 구조를 채택합니다.

## 미구현 / 제약사항

- 현재는 설계 문서와 모델 정의까지 반영된 상태이며, 실제 화면 및 API Mock 구현은 진행 예정입니다.
- drag-and-drop 기반 편집은 범위에서 제외하고, 클릭 및 모달 편집 중심 UX로 구현합니다.
- 차트 기반 주간 요약 시각화는 우선순위가 낮아 후순위로 검토합니다.
- draft는 서버 API 기반으로 설계하며, autosave 주기나 만료 정책은 이번 MVP 범위에서 단순화합니다.
