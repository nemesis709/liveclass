과제 B — 학습 플래너
유형: 시간 기반 UI + 일괄 저장 + 상태 분리

과제 목적
온라인 교육 플랫폼에서 한 주간의 학습 스케줄을 편집하고 저장하는 플래너를 구현합니다. 시간 기반 UI 처리, 시간 로직 설계(충돌 감지·위치 계산), 편집 중(클라이언트) 상태와 저장된(서버) 상태의 분리를 얼마나 안정적이고 일관성 있게 구현하는지를 평가합니다.

배경 시나리오
수강생은 한 주간의 학습 스케줄을 시각적으로 편집합니다.
주간 타임테이블에서 시간 슬롯을 클릭하여 학습 블록을 생성합니다.
각 블록에는 강의를 배정하고 메모를 남길 수 있습니다.
같은 시간대에 겹치는 블록이 있으면 경고를 표시하고 저장을 차단합니다.
편집이 끝나면 "저장" 버튼으로 전체 스케줄을 일괄 저장합니다.
현재 주간의 학습 요약(총 시간, 강의별, 요일별)을 편집 화면 안에서 확인할 수 있습니다.

서버에 저장된 주간 스케줄을 초기 로드

빈 시간 슬롯 클릭 시 블록 추가 모달 표시
등록된 블록 클릭 시 편집/삭제 가능
시간 충돌 감지: 겹치는 블록이 있으면 시각적 경고 표시
주간 요약 섹션: 그리드 상단 또는 하단에 총 학습시간, 강의별 배분, 요일별 학습시간을 숫자·리스트 텍스트로 표시. 편집 중 상태를 실시간 반영 (차트는 선택 구현)
저장 버튼: 현재 편집 상태를 서버에 일괄 저장. 충돌이 있으면 저장 차단

강의 선택 (드롭다운 또는 검색)

메모 (선택, 최대 200자)
모달 확인 시 시간 충돌 검증 → 충돌 시 경고 메시지 (모달 저장 차단)
모달의 "확인"은 편집 중(로컬) 상태에만 반영되며, 서버 반영은 별도 "저장" 버튼으로 수행
저장 결과
성공 시: 편집 상태를 서버 응답으로 동기화, 성공 피드백 표시
실패 시: 에러 메시지 + 재시도 가능 (편집 상태 유지)
선택 구현 (추가 점수)
저장 전 이탈 방지: 변경 사항이 있는 상태에서 페이지 이탈/새로고침 시 확인 대화상자
주간 이동: 이전 주 / 다음 주 네비게이션 (미저장 변경사항이 있으면 이동 전 확인)
반응형 레이아웃: 모바일에서는 일별 뷰 전환
30분 단위 시간 정밀도 지원
주간 요약의 차트·그래프 시각화 (바 차트, 파이 차트 등)

주간 플래너 조회
GET /api/planner?weekStart={ISO 8601}
interface StudyBlock {
  id: string;
  courseId: string;
  dayOfWeek: number;      // 0(월) ~ 6(일) — 주의: JavaScript Date.getDay()는 0(일) 기준이므로 변환 필요
  startTime: string;      // "09:00" (HH:mm)
  endTime: string;        // "10:00" (HH:mm, 30분 단위는 가산점 사양)
  memo?: string;
}

interface PlannerResponse {
  weekStart: string;      // 요청한 주의 월요일
  blocks: StudyBlock[];   // 저장된 블록 (없으면 빈 배열)
}

주간 플래너 저장
PUT /api/planner
interface SavePlannerRequest {
  weekStart: string;
  blocks: Array<{
    id?: string;          // 기존 블록은 id 포함, 신규는 생략 (서버가 생성)
    courseId: string;
    dayOfWeek: number;
    startTime: string;
    endTime: string;
    memo?: string;
  }>;
}

interface SavePlannerResponse {
  weekStart: string;
  blocks: StudyBlock[];   // 저장 후 확정된 블록 (id 포함)
}

면접 연계 질문 (이 과제 전용)
시간 충돌을 어떤 기준으로 판정했나요? 경계값(10:00~11:00과 11:00~12:00)은 충돌인가요?
블록이 100개 이상인 주간 뷰에서 성능 문제가 생긴다면 어떻게 대응하겠나요?
그리드에서 시간 블록의 위치를 어떻게 계산했나요? CSS Grid, absolute positioning, 또는 다른 방식?
서버에 저장된 스케줄과 편집 중인 스케줄을 어떻게 분리 관리했나요? 저장 실패 시 편집 상태를 어떻게 복구하나요?
주간 요약 데이터를 서버에서 계산하는 것과 클라이언트에서 계산하는 것의 trade-off는?
판단 근거 — 이 과제를 선정한 이유
시간 기반 UI는 실무에서 빈번함: 캘린더, 예약, 스케줄 관련 UI는 에드테크에서 핵심 기능이며, 시간 데이터 처리 능력을 자연스럽게 검증
서버 상태 vs 편집 상태 분리 평가: 편집 중 데이터와 저장된 데이터의 구분은 실무에서 빈번한 패턴이며, 상태관리 역량을 변별
과제 A와 성격이 명확히 다름: A는 폼/입력 중심, B는 시각화/시간 로직/상태 분리 중심. 지원자가 강점에 맞게 선택 가능
AI 사용 후 통제 여부 판별 용이: AI는 기본 그리드 화면을 잘 만들지만, 시간 블록의 그리드 배치 정확도, 충돌 감지 엣지 케이스, 편집/저장 상태 분리는 지원자가 직접 확인하고 조정해야 하는 영역
